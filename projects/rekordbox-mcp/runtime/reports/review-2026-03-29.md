## Architecture Summary

The project follows a clean **adapter/domain/service/mcp_tools** layered architecture:

| Layer | Purpose | Quality |
|-------|---------|---------|
| `adapters/` | pyrekordbox + filesystem integration | ✅ Clean boundary |
| `domain/` | Normalized Track and Playlist models | ✅ Well-defined |
| `services/` | Business logic (library, enrichment, playlists, etc.) | ✅ Good separation |
| `mcp_tools/` | Thin MCP transport wrappers | ✅ Transport-agnostic |
| `tools/` | Backwards-compatible shims | ⚠️ Legacy, to deprecate |

**Architecture Strengths:**
- Clear separation of concerns
- Domain models are decoupled from pyrekordbox ORM
- Service layer is transport-agnostic (future FastAPI ready)
- Explicit commit/rollback handling for writes
- Fixture-backed integration tests

**Architecture Weaknesses:**
- `tools/` layer is redundant with `mcp_tools/`—tech debt to consolidate
- Some enrichment logic still in `tools/enrichment.py` rather than services
- No app-owned database yet (depends on live Rekordbox DB)

## Risks

### 1. Live Rekordbox DB Dependency
The system currently requires Rekordbox to be closed for safe writes. Until Phase 4 (app-owned DB) is complete, all enrichment must be dry-run first, then applied carefully.

### 2. MusicBrainz Rate Limits
Batch enrichment can hit MusicBrainz API limits. The current batching (10 per batch) handles this, but scaling to larger libraries needs throttling.

### 3. Scope Creep Before Foundation
The roadmap is ambitious (10 phases). The current priority is to **finish Phase 1** before adding FastAPI or web UI. The enrichment rerun is complete; now lock the defaults and commit.
